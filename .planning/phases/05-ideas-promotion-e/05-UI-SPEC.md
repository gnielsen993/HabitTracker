# Phase 5 ‚Äî UI Design Contract

> Visual and interaction contract for the Ideas + Promotion surfaces (the capture-first spine).
> Generated by gsd-ui-researcher from `05-CONTEXT.md` decisions D-01‚Ä¶D-14, verified against
> DesignKit source files, and authored in the same idiom as the approved Phase 2
> (`02-UI-SPEC.md`) and Phase 4 (`04-UI-SPEC.md`) contracts.
>
> **Platform note:** Native **SwiftUI / iOS** backed by the shared **DesignKit** Swift Package ‚Äî
> NOT web/CSS. No shadcn, no registry, no hex values. All values are DesignKit semantic tokens
> and SF Symbols. `CLAUDE.md` hard rules are binding: no hard-coded colors, Balanced Luxury
> restraint, spacing/radii/typography from DesignKit tokens only.
>
> **Continuity:** This contract reuses the Phase 1‚Äì4 token vocabulary verbatim, the section-header
> "+" pattern (Phase 2 D-12 / Phase 3 D-09 / Phase 4 D-10), and `accentColor(forToken:)`. It also
> **deliberately breaks** one precedent on purpose: Ideas get NO detail view and NO heavyweight
> editor (D-08) ‚Äî this is the one type that opts out of the row‚Üídetail nav template, by design.
>
> **Capture-first gate (binding on every surface below):** the global quick-add must be reachable
> without leaving Today and must NOT add a row to Today's list (IDEA-02, `DEC-today-is-hero`). No
> network fetch anywhere in this phase (ecosystem offline-only constraint carries over unchanged).

## Design System

| Property | Value |
|----------|-------|

## Design System Exceptions

Held over verbatim from the approved Phase 4 contract ‚Äî same underlying token system, no new deviations introduced this phase.

### Typography: up to 6 roles / DesignKit-fixed weights

**Exception:** The generic checker ceiling of "‚â§4 roles / ‚â§2 weights" targets ad-hoc CSS type scales, not an established centralized native design system.

**Rationale:** `DesignKit/Typography/TypographyTokens.swift` defines exactly **6 semantic roles** (`titleLarge`, `title`, `headline`, `body`, `caption`, `monoNumber`). Phase 5 introduces **zero new sizes and zero new weights** ‚Äî it uses 4 of the 6 roles (no `titleLarge` ‚Äî Ideas has no hero empty-state moment of its own this phase per D-04; no `monoNumber` ‚Äî no numeric rollup).

| Surface | `title` | `headline` | `body` | `caption` |
|---------|:---:|:---:|:---:|:---:|
| S3 ‚Äî Inbox card | ‚Äî | "Ideas to file" | ‚Äî | ‚Äî (count lives in `DKBadge`, which applies `caption.weight(.semibold)`) |
| S4 ‚Äî InboxView | ‚Äî | ‚Äî | minimal placeholder text | ‚Äî |
| S5 ‚Äî Ideas section header | section title "Ideas" | ‚Äî | ‚Äî | ‚Äî |
| S6 ‚Äî IdeaRow | ‚Äî | idea title | ‚Äî | note preview; File/Promote pill labels (via the `DKBadge`-style recipe) |
| S2 ‚Äî IdeaCaptureSheet | ‚Äî | field label | title field value | validation hint |
| S7 ‚Äî PromoteToCollectionPicker | ‚Äî | collection name | ‚Äî | domain-name sub-label |

### Spacing: `theme.spacing.m = 12`

**Exception:** The checker flags `m = 12` as off the "standard 4-8-16-24-32-48-64" set. Correct only for web/CSS scales.

**Rationale:** `DesignKit/Layout/SpacingTokens.swift` defines `m = 12` as a first-class semantic token (verified: `xs=4, s=8, m=12, l=16, xl=24, xxl=32`), on the 4pt grid (12 = 3√ó4). Used app-wide in Phases 1‚Äì4; Phase 5 reuses it between the File and Promote pills on `IdeaRow`.

## Color

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `theme.colors.background` | Screen background for `InboxView`, `IdeaCaptureSheet`, `PromoteToCollectionPicker` |
| Secondary (30%) | `theme.colors.surface` | Inbox card, `IdeaRow`, `PromoteToCollectionPicker` rows (all `DKCard` surfaces) |
| Border | `theme.colors.border` | Row/card stroke (1pt) ‚Äî inherited from `DKCard` |
| Text ‚Äî primary | `theme.colors.textPrimary` | Idea titles; inbox-card title; capture-sheet field values |
| Text ‚Äî secondary | `theme.colors.textSecondary` | Note preview; placeholder text; validation hint; `PromoteToCollectionPicker` domain sub-label |
| Text ‚Äî tertiary | `theme.colors.textTertiary` | Chevron on the inbox card |
| Accent ‚Äî theme | `theme.colors.accentPrimary` | Global "+" toolbar item; Ideas section "+" button; File pill; Promote pill; inbox-card leading icon; capture-sheet Save CTA |
| Highlight | `theme.colors.highlight` | File pill / Promote pill background (`DKBadge`-style recipe); inbox-card `DKBadge` count background |
| Destructive | `theme.colors.danger` | "Delete Idea" row + confirmation confirm button (edit-mode capture sheet) |

**Accent (`accentColor(forToken:)`, the per-domain resolver) is NOT used on any Idea-specific surface.** This is a deliberate departure from Rules/Collections/Clips (which tint a domain glyph in their detail headers): an idea in the Inbox is by definition unfiled, so it has no domain to tint; an idea inside a domain's Ideas section already sits under that domain's own header (which already shows the domain glyph, untouched by this phase) ‚Äî repeating the domain color on every row would be redundant, not reinforcing. Ideas stay visually domain-agnostic in their own chrome; only `theme.colors.accentPrimary` (the generic theme accent) is used for interactive elements, consistent with how "Stem habit" / "Open Link" CTAs use `accentPrimary` rather than a domain color in Phases 2 and 4.

**File pill and Promote pill color semantics:** both render through the same `DKBadge`-style token pairing (`accentPrimary` text on `highlight` background) as the Clips status chip and tag pill ‚Äî visual consistency across all row-level pill affordances in the app, no per-action color coding.

## Component Inventory & Interaction Contract

### S1 ‚Äî Global quick-add "+" (IDEA-02, D-01, D-02)

- `TodayView`'s `NavigationStack` currently has **no toolbar** ‚Äî this is net-new chrome, added without touching any existing Today content.
- `.toolbar { ToolbarItem(placement: .topBarTrailing) { Button { showingCapture = true } label: { Image(systemName: "plus") } } }`. Icon styled `.font(.system(size: 18, weight: .semibold))`, `theme.colors.accentPrimary` tint (matches the section-header "+" glyph treatment exactly ‚Äî one consistent "+" visual language app-wide). ‚â•44pt tap target (toolbar items get this natively).
- Presents `IdeaCaptureSheet()` (S2, no domain param ‚Üí unfiled) via `.sheet(isPresented: $showingCapture)`.
- **Does not add a row to Today's list** ‚Äî this is purely nav-bar chrome; Today's `ScrollView`/`VStack` content is untouched.
- `accessibilityLabel`: **"Add idea"**. `accessibilityHint`: **"Opens a quick capture sheet"**.

### S2 ‚Äî `IdeaCaptureSheet` (IDEA-01, IDEA-02, D-08)

The **one shared component** for both the global quick-add and the in-domain Ideas "+" (S5), and for editing an existing idea's title (tap on an `IdeaRow`, excluding its File/Promote pills). Genuinely **title-only** per D-08's literal instruction ‚Äî no note/url/domain fields exposed in this sheet. (The `Idea` model's optional `note`/`url` fields exist for promote-carry plumbing per D-11 ‚Äî this phase's UI never writes to them directly; see Open Items.)

- Presented as a **sheet** (`theme.radii.sheet`), `NavigationStack` + minimal `Form` or plain `VStack` ‚Äî deliberately lighter than `RuleEditorView`/`ClipEditorView` (single field, not a multi-section Form).
- **Signature:** `IdeaCaptureSheet(domain: Domain? = nil)` for create (global quick-add passes `nil`; in-domain "+" passes the owning domain ‚Äî pre-filed, no domain picker shown either way, matching D-09's "place-first" in-domain entry). `IdeaCaptureSheet(idea: Idea)` for edit (title prefilled).
- **Field:** one `TextField("What's the idea?", text: $title)`, `theme.typography.body` value, autofocus on appear (`@FocusState`, matches the "near-zero friction" capture-first framing ‚Äî the keyboard should already be up). Header/label above the field: **"Title"** (`theme.typography.headline`).
- Below the field, when `trimmedTitle.isEmpty`: hint **"Give this a name to continue."** (`theme.typography.caption`, textSecondary) ‚Äî verbatim reuse from Rule/Clip editors.
- **Save CTA:** **"Add Idea"** (create) / **"Save Changes"** (edit). Toolbar `.confirmationAction`, disabled until `trimmedTitle` non-empty. On save: create-mode inserts `Idea(title:, domain:)` + `try modelContext.save()`; edit-mode mutates `title` in place + saves.
- **Cancel:** toolbar `.cancellationAction`, **"Cancel"** ‚Äî dismisses, creates/changes nothing (fill-then-commit, matches `HabitCreateSheet`'s orphan-free pattern).
- **Delete (edit mode only):** a destructive **"Delete Idea"** row (`theme.colors.danger`), `Button(role:.destructive)`. Confirmation: `confirmationDialog` ‚Äî title **"Delete this idea?"**, message **"This can't be undone."**, confirm **"Delete Idea"** (danger), cancel **"Cancel"** (matches the Clips delete-confirm shape exactly). This is a hard delete (not archive-with-forward-link ‚Äî that's reserved for Promote, D-07); it's the escape hatch for a mis-captured idea.
- **Accessibility:** `accessibilityLabel` on the field: **"Idea title"**. Sheet `navigationTitle`: **"New Idea"** (create) / **"Edit Idea"** (edit).

### S3 ‚Äî Hub inbox card (IDEA-03, D-03, D-04)

- Pinned in `HubView`, **above** `grid(theme:)`, inside the same `ScrollView`/`VStack` that currently renders the domain grid ‚Äî not a separate screen region.
- **Visibility (D-04):** rendered only when `@Query`-derived unfiled (`domain == nil`), non-archived idea count > 0. Zero ideas ‚Üí the card is entirely absent (no ghost/zero state ‚Äî Phase 6/POL-02 owns the designed empty state).
- **Layout:** `DKCard` surface, `HStack(spacing: theme.spacing.s)`:
  - Leading `Image(systemName: "tray.full")`, `theme.colors.accentPrimary` tint.
  - `Text("Ideas to file")`, `theme.typography.headline`, `theme.colors.textPrimary`.
  - `Spacer()`.
  - Trailing `DKBadge("\(count) to file", theme: theme)` ‚Äî reuses the existing chip component verbatim, no new pill recipe.
  - Trailing `Image(systemName: "chevron.right")`, `theme.colors.textTertiary`, small.
- Full card is one ‚â•44pt tap target, `NavigationLink` ‚Üí `InboxView` (S4), `.buttonStyle(.plain)`.
- **Copy for N:** use `"{N} to file"` verbatim for every N including 1 (no singular/plural special-casing this phase ‚Äî acceptable per D-04's "minimal placeholder" posture; Phase 6 may refine).
- `accessibilityLabel`: **"{N} ideas to file, opens inbox"**.

### S4 ‚Äî `InboxView` (IDEA-03, D-05)

- Data-driven per ¬ß9.2: owns its own `@Query(filter: #Predicate<Idea> { $0.domain == nil && !$0.isArchived }, sort: \Idea.createdAt, order: .reverse)` (recency-first ‚Äî matches the capture-first "most recent capture first" framing; also matches Rule/Clip row ordering precedent).
- `navigationTitle`: **"Inbox"**. Declares its own `NavigationStack`? **No** ‚Äî pushed from `HubView`'s existing stack via the S3 `NavigationLink`, so it nests (matches `DomainDetailView`'s pattern of not owning a second nav container).
- **List:** `ScrollView` + `VStack(spacing: theme.spacing.m)` of `IdeaRow` (S6), each fed one `Idea` value, both File and Promote pills shown (every row here is unfiled by the query's own definition).
- **Minimal empty state (D-04 ‚Äî NOT the Phase 6 designed empty state):** if the query transitions to zero while the view is open (last idea filed/promoted), show a single centered line: **"Nothing to file right now."** (`theme.typography.body`, `theme.colors.textSecondary`), `theme.spacing.xxl` top padding, no heading, no illustration, no CTA. Flagged explicitly: **POL-02 replaces this.**

### S5 ‚Äî Ideas section in `DomainDetailView` (IDEA-01, IDEA-03, D-09)

- **Integration:** append a `DomainSection(id:"ideas", title:"Ideas", ‚Ä¶)` to `nonEmptySections`, replacing the "// Phase E: append Ideas section here" hook at ~line 90. Follows the exact `buildClipsSection` / `clipsSectionContent` / `clipsSectionHeader` trio shape (D-09's explicit instruction to keep the per-section "+" pattern; do not build a unified domain-level "+" picker).
- **Section header:** `Text("Ideas")` styled `theme.typography.title`, `.accessibilityAddTraits(.isHeader)`, plus a trailing **"+"** button (`Image(systemName:"plus")`, `.font(.system(size: 18, weight: .semibold))`, `accentPrimary` tint, ‚â•44pt) that presents `IdeaCaptureSheet(domain: domain)` (S2) as a sheet, pre-filed under `domain` (place-first, no domain picker shown). `accessibilityLabel`: **"Add idea to \(domain.name)"**.
- **Section visibility:** renders only when `domain.ideas` has at least one non-archived, filed (i.e. `domain != nil` is implicit ‚Äî these are already scoped to this domain) idea ‚Äî a domain with zero ideas shows no Ideas section (mirrors D-10's Rules/Collections/Clips contract exactly).
- **Rows:** `IdeaRow` (S6), fed each filed idea. Because `idea.domain != nil` here, the File pill is **absent** on these rows (derived automatically from the idea's own `domain` field ‚Äî see S6); only the Promote pill shows.
- **Ordering:** ideas sorted by `createdAt` descending (matches Rules/Clips ‚Äî recency-first).
- No swipe actions on the row (matches Phase 2/3/4 precedent ‚Äî this app reserves `.swipeActions` for destructive-only affordances elsewhere, e.g. `DomainFocusPicker`'s custom-domain delete; File/Promote are graduation actions, not deletions, and stay visible inline per D-08's own wording).

### S6 ‚Äî `IdeaRow` (reusable, data-driven per ¬ß9.2) (IDEA-03, D-05, D-10)

Shared between S4 (`InboxView`) and S5 (Ideas section) ‚Äî takes only an `Idea` value, owns no `@Query`, no `modelContext` access beyond the two mutation closures it's handed (or performs directly via `@Environment(\.modelContext)`, consistent with `ClipRow`'s status-chip toggle pattern).

- **Layout:** `DKCard` surface, `VStack(alignment: .leading, spacing: theme.spacing.s)`:
  - **Row 1 (tap ‚Üí edit title, S2 in edit mode):** `Text(idea.title)`, `theme.typography.headline`, `theme.colors.textPrimary`, `.lineLimit(2)`, `.minimumScaleFactor(0.9)` (matches `RuleRow`/`ClipRow`). If `idea.note` is present and non-empty, a single-line truncated caption below in `theme.typography.caption`/textSecondary (visual parity with `ClipRow`'s optional tag line ‚Äî even though S2 never writes `note` this phase, the row still renders it defensively for forward-compat with D-11's schema).
  - **Row 2 (actions):** `HStack(spacing: theme.spacing.s)`:
    - **File pill** (`Image(systemName:"tray.and.arrow.down")` + `Text("File")`) inside a `Menu` listing all domains (`@Query(sort: \Domain.sortIndex)`, owned by the row per the "only mutation closures" exception above, or passed in ‚Äî planner's call) ‚Äî each `Button(domain.name) { idea.domain = domain; try? modelContext.save() }`. **Shown only when `idea.domain == nil`** ‚Äî derived straight from the value, no parent-side flag needed. Pill styling: `DKBadge`-equivalent recipe (`caption.weight(.semibold)` on `accentPrimary`/`highlight`, `theme.radii.chip`, ‚â•44pt via `.frame(minHeight:44)` + `.contentShape(Rectangle())`).
    - **Promote pill** (`Image(systemName:"arrow.up.forward.circle")` + `Text("Promote")`) inside a `Menu` with three items ‚Äî `Button("Rule")`, `Button("Habit")`, `Button("Collection item")` ‚Äî routing per S7. **Always shown** (filed or unfiled ideas can both promote, D-07). Same pill styling as File.
  - Tapping anywhere on Row 1 (title/note text, NOT the action row) opens `IdeaCaptureSheet(idea: idea)` (S2, edit mode) ‚Äî this is the row's only "detail" affordance, deliberately shallow (D-08).
- **Accessibility:** `.accessibilityElement(children: .combine)` on Row 1 with label **"\(idea.title), idea"** (+ note text if present); the File and Promote `Menu`s remain separate, individually reachable VoiceOver elements (not absorbed into the combined label) ‚Äî `accessibilityLabel` **"File idea, choose a domain"** / **"Promote idea, choose a type"** respectively.

### S7 ‚Äî Promote routing (IDEA-04, IDEA-05, D-06, D-07)

Promote is **menu ‚Üí existing prefilled editor ‚Üí Save**, never a bespoke `PromoteSheet` (D-06). All three routes share one small, testable **promote consume/archive/forward-link helper** (service, not scattered in views, ¬ß9.5) ‚Äî invoked from each target editor's Save action only when that editor was launched via promote (vs. its normal create flow).

- **"Rule"** ‚Üí presents `RuleEditorView` prefilled: `title = idea.title`, `body = idea.note ?? ""`, `sourceURL = idea.url`, `domain = idea.domain` (nil if unfiled). **Domain-required save-gate:** when launched via promote AND `idea.domain == nil`, `RuleEditorView`'s existing domain `Picker` must have a selection before Save enables (reuse the picker, add the gate ‚Äî no new UI, per D-07's literal instruction). A filed idea's promote defaults to its own domain and needs no extra gate.
- **"Habit"** ‚Üí presents `HabitCreateSheet(source: .idea(idea))` (add the `.idea(Idea)` case to `HabitSource` ‚Äî already reserved in code with the comment "Phase 5 will add `.idea(Idea)` without touching the sheet"). Chrome and fields are **completely untouched** (D-07) ‚Äî title/domain prefill from the idea, both remain editable, exactly like the existing `.rule(Rule)` case. No domain-required gate here (Habit's domain has always been optional/editable in this sheet, unlike Rule/Collection).
- **"Collection item"** ‚Üí presents `PromoteToCollectionPicker` (new, lightweight sheet) **first** ‚Äî IDEA-05's "promote-to-collection prompts for the target list":
  - `navigationTitle`: **"Choose a List"**. Plain `List`/`ScrollView` of `DKCard` rows, one per `Collection` app-wide, `theme.spacing.m` between rows: collection name (`theme.typography.headline`, textPrimary) + owning domain name as a caption sub-label (`theme.typography.caption`, textSecondary) beneath it (disambiguates same-named lists across domains).
  - Tapping a row dismisses this picker and opens that collection's item editor, prefilled `title = idea.title`; `url`/cost fields carry if the target editor has matching fields and `idea.url` is present.
  - **Domain-required save-gate:** if reached via an unfiled idea, the collection itself already implies a domain (a `Collection` always belongs to one) ‚Äî no additional domain picker is needed in the item editor; selecting the collection *is* the domain selection.
  - **Empty-collections edge case:** if zero collections exist app-wide, show a single centered line **"No lists yet. Create a collection first."** (`theme.typography.body`, textSecondary) ‚Äî minimal, matches the D-04 minimal-placeholder posture elsewhere this phase.
  - Toolbar `.cancellationAction`: **"Cancel"** ‚Äî dismisses without promoting.
- **On any successful Save reached via promote (all three routes):** the promote helper sets `idea.isArchived = true` + the forward-link fields (D-12, exact shape is schema-layer discretion, out of this UI-SPEC's scope), and the idea disappears from whichever list is currently showing it (Inbox or the domain's Ideas section) since both queries filter `!isArchived`. **The result carries no backref to the idea** (D-07) ‚Äî no "Promoted from" chip is shown anywhere on the new Rule/Habit/CollectionItem.

### Accessibility (¬ß9.15 ‚Äî part of "done", applies to all surfaces)

- **Global "+" (S1):** `accessibilityLabel` "Add idea"; `accessibilityHint` "Opens a quick capture sheet".
- **Inbox card (S3):** `accessibilityLabel` "{N} ideas to file, opens inbox".
- **Idea title field (S2):** `accessibilityLabel` "Idea title".
- **Idea row (S6):** combined `accessibilityLabel` "{title}, idea" (+ note when present); File/Promote pills remain individually reachable with their own labels (never absorbed into the row's combined element ‚Äî VoiceOver users must be able to reach both actions independently, matching `ClipRow`'s "status chip stays separate from combined text" precedent).

## Open Items for Planner/Executor (Claude's Discretion carried from CONTEXT.md)

These are explicitly left open by `05-CONTEXT.md` and are NOT blocking this UI-SPEC ‚Äî they are scoped decision points the planner/executor resolves within the constraints stated above:

1. **`Idea.note`/`Idea.url` are model-level plumbing this phase's UI never writes to.** They exist per D-11 so promote-carries work (Rule `sourceURL`, future Clip-style carries), but `IdeaCaptureSheet` is genuinely title-only (D-08 literal). Do not add note/url fields to the capture sheet ‚Äî that would contradict D-08 and the "near-zero friction" spec intent. If a future phase wants to expose them, that's a new UI-SPEC decision, not an oversight here.
2. Exact forward-link field shape on `Idea` (D-12) ‚Äî schema-layer, not a visual-contract concern; this spec only requires that a promoted idea vanishes from active lists and the target carries no backref.
3. Whether the promote consume/archive/forward-link logic lives in a dedicated `PromoteService.swift` vs. a smaller inline helper ‚Äî either satisfies ¬ß9.5 as long as it's pure/testable and centralized (not scattered across the three target editors).
4. Whether `IdeaRow`'s File/Promote `Menu`s own their own `@Query(sort: \Domain.sortIndex)` / collection query directly, or receive the lists as parameters from the parent (`InboxView`/`DomainDetailView`) ‚Äî either is compatible with ¬ß9.2's "data-driven" intent since the row still takes its core `Idea` value as a prop; the query is auxiliary picker data, not the row's primary subject.
5. `RuleEditorView`'s and the collection-item editor's exact mechanism for the "domain-required when reached via promote from an unfiled idea" save-gate (e.g. an `isPromoteFlow: Bool` init parameter vs. inferring from a passed-in `sourceIdea: Idea?`) ‚Äî implementation detail, not a visual one; the contract only requires Save stays disabled until a domain is set in that specific path.
6. `PromoteToCollectionPicker`'s exact grouping (flat list with domain sub-label, per this spec, vs. `Section`-grouped by domain) ‚Äî flat list with sub-label is the baseline; sectioned grouping is an acceptable upgrade if the collection count warrants it, still within tokens.
