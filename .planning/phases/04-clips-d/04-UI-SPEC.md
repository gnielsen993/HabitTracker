# Phase 4 — UI Design Contract

> Visual and interaction contract for the Clips surfaces. Generated by gsd-ui-researcher from 04-CONTEXT.md decisions D-01…D-13, verified against DesignKit source files, and authored in the same idiom as the approved Phase 2 (02-UI-SPEC.md) and Phase 3 (03-UI-SPEC.md) contracts.
>
> **Platform note:** Native **SwiftUI / iOS** backed by the shared **DesignKit** Swift Package — NOT web/CSS. No shadcn, no registry, no hex values. All values are DesignKit semantic tokens and SF Symbols. `CLAUDE.md` hard rules are binding: no hard-coded colors, accents constrained to the domain-token set, Balanced Luxury restraint, spacing/radii/typography from DesignKit tokens only.
>
> **Continuity:** This contract reuses the Phase 1/2/3 token vocabulary verbatim. The nav shape (domain section → "+" in section header → editor → row → detail view) mirrors Phase 2's D-12 / Phase 3's D-09 template exactly. `accentColor(forToken:)` from Phase 1 and the `DKBadge` status-chip idiom from Phase 3 (D-04 here) are consumed without change.
>
> **Offline gate (SC1 / D-01, binding on every surface below):** Clips are fully offline. No network fetch, ever — no rich preview, no thumbnail/metadata retrieval. `ClipEditorView`'s title suggestion (D-02) is a pure, zero-network string parser only.

## Design System Exceptions

This section documents deliberate deviations from generic checker heuristics. These are not violations — they are the correct application of the established native design system, held over verbatim from the approved Phase 3 contract.

### Typography: up to 6 roles / DesignKit-fixed weights

**Exception:** The generic checker ceiling of "≤4 roles / ≤2 weights" is a heuristic aimed at ad-hoc CSS type scales. It does not govern an established, centralized native design system.

**Rationale:** `DesignKit/Typography/TypographyTokens.swift` defines exactly **6 semantic roles** (`titleLarge`, `title`, `headline`, `body`, `caption`, `monoNumber`). Phase 4 introduces **zero new sizes and zero new weights** — it uses 4 of the 6 roles (no `titleLarge`, no `monoNumber` — Clips has no empty-state hero moment of its own beyond the shared domain-level empty state, and no numeric rollup).

| Surface | `title` | `headline` | `body` | `caption` |
|---------|:---:|:---:|:---:|:---:|
| S1 — ClipRow | — | clip title | — | tag / status sub-label |
| S2 — ClipDetailView | clip title | block labels ("Note", "Tag") | note content; Open Link host text | tag value; full URL; status chip label (via `DKBadge`) |
| S3 — ClipEditorView | — | field labels | field values; suggested-title hint | validation hint |

### Spacing: `theme.spacing.m = 12`

**Exception:** The checker flags `m = 12` as off the "standard 4-8-16-24-32-48-64" set. Correct only for web/CSS scales.

**Rationale:** `DesignKit/Layout/SpacingTokens.swift` defines `m = 12` as a first-class semantic token (verified: `xs=4, s=8, m=12, l=16, xl=24, xxl=32`), on the 4pt grid (12 = 3×4). Used app-wide in Phases 1–3; Phase 4 reuses it between the status chip and tag on `ClipRow`, and between blocks in `ClipDetailView`.

## Spacing Scale

Source of truth: `DesignKit/Layout/SpacingTokens.swift` (verified: `xs=4, s=8, m=12, l=16, xl=24, xxl=32`). **Do not invent values — use these tokens by name.**

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| `theme.spacing.xs` | 4 | Chip label horizontal inset (`DKBadge` internal); icon ↔ text gap on Open Link affordance; tag pill inset |
| `theme.spacing.s` | 8 | Between status chip and tag on `ClipRow`; between form field label and value; Open Link affordance inner spacing |
| `theme.spacing.m` | 12 | Between `ClipRow` entries; between blocks in `ClipDetailView` (status ↔ tag ↔ note ↔ Open Link) |
| `theme.spacing.l` | 16 | `DKCard` surface padding; section content inset; sheet horizontal padding |
| `theme.spacing.xl` | 24 | Between major blocks in `ClipDetailView` (header → status/tag → note → Open Link) |
| `theme.spacing.xxl` | 32 | Empty-state top inset (shared domain-level empty state; no Clips-specific empty state — see Copywriting) |

Radii — source `DesignKit/Layout/RadiusTokens.swift` (verified: `card=16, button=14, chip=12, sheet=22`):

| Token | Value | Usage |
|-------|-------|-------|
| `theme.radii.card` | 16 | `ClipRow`, Open Link affordance card in `ClipDetailView` |
| `theme.radii.button` | 14 | "Open Link" primary CTA; "Save"/"Add Clip" CTA |
| `theme.radii.chip` | 12 | Status chip (`DKBadge`); tag pill |
| `theme.radii.sheet` | 22 | `ClipEditorView` sheet surface |

**Exceptions:** All tap targets (clip rows, status chip, Open Link affordance, "+" header button, Edit button) must be **≥ 44×44 pt** via `.frame(minWidth:44, minHeight:44)` — the only allowed non-token dimension (accessibility floor). Matches Phases 1–3.

## Typography

Source of truth: `DesignKit/Typography/TypographyTokens.swift` (verified — 6 token roles). Do not introduce new `.font(.system(size:))` calls except the existing house exception for section-header "+" glyphs (`.font(.system(size: 18, weight: .semibold))`, already used in `DomainDetailView` for Rules/Collections — Clips reuses this exact pattern, not a new one).

| Role | Token | Underlying | Usage in this phase |
|------|-------|-----------|---------------------|
| Heading | `theme.typography.title` | title2 / semibold | Section header "Clips"; `ClipDetailView` clip title |
| Label | `theme.typography.headline` | headline | `ClipRow` clip title; `ClipDetailView` block labels ("Note", "Tag"); form field labels |
| Body | `theme.typography.body` | body / regular | `ClipDetailView` note content, Open Link host text; form field values; suggested-title hint |
| Caption | `theme.typography.caption` | caption | `ClipRow` tag/status sub-label; `ClipDetailView` tag value, full URL under host; validation hint; status chip label uses `DKBadge` which applies `caption.weight(.semibold)` |

- Dynamic Type: all text scales; `ClipRow` title caps at 2 lines with `.minimumScaleFactor(0.9)` (matches `RuleRow`). All other text wraps freely.
- Line height: SwiftUI defaults per token; do not override.
- `DKBadge` renders `caption.weight(.semibold)` with `accentPrimary` foreground on `highlight` background — consume as-is for the status chip AND the tag pill (D-04's "same DesignKit chip styling" applies to both — see Color section).

## Color

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `theme.colors.background` | Screen background for `ClipDetailView`, sheets |
| Secondary (30%) | `theme.colors.surface` | `ClipRow`, Open Link affordance card |
| Border | `theme.colors.border` | Row/card stroke (1pt); Open Link affordance outline |
| Text — primary | `theme.colors.textPrimary` | Clip titles; note content; Open Link host text; block labels' values |
| Text — secondary | `theme.colors.textSecondary` | Captions; tag sub-label; full URL under host; validation hint |
| Accent (10%) — domain | `accentColor(forToken: domain.colorToken, scheme:)` | **Reserved for:** the domain glyph tint, if `ClipDetailView` chooses to show one (optional — Clips has no per-item domain glyph header requirement; omit unless the domain identity needs reinforcing, matching `RuleDetailView`'s pattern) |
| Accent — theme | `theme.colors.accentPrimary` | "+" section header button; "Open Link" CTA background; "Save"/"Add Clip" CTA; Edit button; `DKBadge` foreground (status chip, tag pill) |
| Highlight | `theme.colors.highlight` | `DKBadge` background (status chip, tag pill) |
| Destructive | `theme.colors.danger` | "Delete Clip" destructive action label (if planner confirms a delete confirm is warranted per D-11) |

**Accent (`accentColor(forToken:)`) reserved for (exhaustive):**
1. The owning domain's glyph in `ClipDetailView`'s header, if shown (mirrors `RuleDetailView` — optional, at executor's layout discretion).

**Accent is NOT used for:** status chip foreground (uses `accentPrimary` via `DKBadge`), tag pill foreground (same), row highlights, or any text — those use `textPrimary` / `accentPrimary` / `danger`. The domain accents remain identity-only (one color = one domain).

**Status chip color semantics:** both states (`saved`, `acted`) render through the same `DKBadge` token pairing (`accentPrimary` text on `highlight` background) — no per-state color override. The chip's text label alone communicates state ("Saved" / "Acted"). This matches the Phase 3 precedent (no color-coding state).

**Tag pill color semantics:** the optional `tag` renders via the same `DKBadge` styling as the status chip (D-04's "visual consistency without coupling" instruction) — `accentPrimary` on `highlight`, no per-tag color. This is deliberately identity-agnostic: a free-text tag is not a taxonomy with assigned colors (PROJECT.md Out of Scope — cross-domain tag system deferred).

## Component Inventory & Interaction Contract

### S1 — Clips section in `DomainDetailView` (CLIP-03)

- **Integration:** append a `DomainSection(id:"clips", title:"Clips", …)` to `nonEmptySections`, in the "Phase D–E: append Clips / Ideas sections here" hook (~line 81). Follows the exact `buildCollectionsSection` / `collectionsSectionContent` / `collectionsSectionHeader` trio shape (D-10).
- **Section header:** `Text("Clips")` styled as `theme.typography.title` with `.accessibilityAddTraits(.isHeader)`, plus a trailing **"+"** button (`Image(systemName:"plus")`, `.font(.system(size: 18, weight: .semibold))`, `accentPrimary` tint, ≥44pt) that presents `ClipEditorView(domain:)` (S3) as a sheet, pre-scoped to `domain`. `accessibilityLabel`: `"Add clip to \(domain.name)"`.
- **Section visibility (D-10):** renders **only when `domain.clips` has at least one non-archived clip** — a domain with zero non-archived clips shows no Clips section (mirrors D-15's Rules/Collections contract exactly).
- **Row (`ClipRow`, data-driven per §9.2):** `DKCard` surface. Layout (`HStack`):
  - Leading `VStack(alignment:.leading, spacing: theme.spacing.xs)`: clip **title** in `theme.typography.headline` (textPrimary), 2-line cap + `.minimumScaleFactor(0.9)` (matches `RuleRow`); optional caption line below showing the `tag` when present, in `theme.typography.caption` (textSecondary) — e.g. `"recipe"`.
  - Spacer.
  - Trailing: **status chip** — `DKBadge(statusLabel, theme: theme)` where `statusLabel` is `"Saved"` or `"Acted"` (D-08: "the status chip also renders on the row … for at-a-glance scannability"). The chip is a tap target:
    - `.onTapGesture`: toggles `status` between `.saved` and `.acted` (D-05 — two-way toggle; see S2 for the identical control). `.sensoryFeedback(.impact(.light))` fires on every tap.
    - Chip min tap target: ≥44pt via `.frame(minWidth:44, minHeight:44)`.
- **Ordering:** clips sorted by `createdAt` descending (most recently saved first — matches `RuleRow`'s ordering; a clip is a "did I save this?" list, recency-first serves the "found where you'd look" goal).
- **Swipe actions:** none on the row (matches Phase 2/3 precedent — delete/archive live in the editor/detail action, not swipe).
- Full row is a `NavigationLink` → push `ClipDetailView(clip:)`, `.buttonStyle(.plain)` (matches Rules/Collections row wiring).

### S2 — `ClipDetailView` (CLIP-02, CLIP-03, CLIP-04)

**Primary focal element:** The **"Open Link" CTA** is the primary focal element (D-08 — "the URL is the payload… made prominent… to preserve the act-on-it feel"), placed directly beneath the title/status block, ahead of the note.

- Data-driven: takes a `Clip` value; declares **no `NavigationStack`** (nests under the Hub stack, matches `RuleDetailView`). Vertical `ScrollView` with `VStack(alignment:.leading, spacing: theme.spacing.xl)`.
- **Block 1 — Header (always shown):**
  - Clip **title** in `theme.typography.title` (textPrimary).
  - Optional domain glyph (accent-tinted via `accentColor(forToken:)`) leading the title, at executor's discretion — mirrors `RuleDetailView`'s header block shape exactly if included.
- **Block 2 — Status + Tag (always shown for status; tag conditional):**
  - **Status chip:** `DKBadge(statusLabel, theme: theme)` ("Saved" / "Acted"). Tap-to-toggle identical to S1 (D-05: plain two-way toggle, `.sensoryFeedback(.impact(.light))` on every tap).
  - **Tag pill** (conditional — omit when `tag` is nil/empty): `DKBadge(tag, theme: theme)` immediately beside or below the status chip, `theme.spacing.s` gap.
- **Block 3 — Note (conditional — omit when `note` is nil/empty):**
  - Label `"Note"` in `theme.typography.headline` (textSecondary), value below in `theme.typography.body` (textPrimary), `.textSelection(.enabled)` (matches `RuleDetailView`'s body block).
- **Block 4 — Open Link (always shown — CLIP-02's `url` is required):**
  - Bordered affordance identical to `RuleDetailView`'s source block pattern: `Image(systemName:"link")` + host (`theme.typography.body`, textPrimary) + full URL (`theme.typography.caption`, textSecondary) inside a `DKCard`-radius (`theme.radii.card`) bordered surface (`theme.colors.surface` background, `theme.colors.border` 1pt stroke). Tap opens via `Link`/`openURL` in Safari (D-08). `accessibilityLabel`: `"Open link, \(host)"`. ≥44pt. Offline-safe (stored + opened only — never fetched).
  - This block is visually promoted above Note (D-08 primacy) — order in the `VStack` is Header → Status/Tag → **Open Link** → Note, OR Header → Status/Tag → Note → Open Link with Open Link rendered as a full-width primary-styled CTA button (`accentPrimary` background, `theme.colors.background` foreground text, `theme.radii.button`, ≥44pt — matching `RuleDetailView`'s "Stem habit" CTA treatment) rather than a plain bordered card. **Use the CTA-button treatment** (not the plain bordered-link block) to fully satisfy D-08's "prominently features" instruction — this is the one deliberate visual upgrade over the Rules source-block precedent, since Clips' entire purpose is act-on-the-link.
- **Toolbar:** trailing **"Edit"** button (`theme.colors.accentPrimary`, `theme.typography.headline` weight via toolbar button styling) → presents `ClipEditorView(clip:)` (S3) as a sheet, matching `RuleDetailView`'s toolbar wiring exactly.
- **Accessibility:** `.accessibilityElement(children:.combine)` on the header block; `accessibilityLabel`: `"\(clip.title), clip, status: \(statusLabel)\(tagSuffix)"`.

### S3 — `ClipEditorView` (CLIP-01, CLIP-02)

Presented as a **sheet** (`theme.radii.sheet`), modeled directly after `RuleEditorView` — `NavigationStack` + `Form`, `.scrollContentBackground(.hidden)`, `theme.colors.background`. Two modes:
- **Create:** `ClipEditorView(domain:)` — inserts a new `Clip`, pre-filed under `domain`, defaults `status = .saved`. Save CTA: **"Add Clip"**.
- **Edit:** `ClipEditorView(clip:)` — mutates in place. Save CTA: **"Save Changes"**.

**Fields (Form sections, in order):**
1. **URL** (`TextField("https://example.com", text: $urlText)`, `.keyboardType(.URL)`, `.textInputAutocapitalization(.never)`, `.autocorrectionDisabled()`) — required. Header: `"URL"` (`theme.typography.headline`).
   - **D-02 title-suggestion wiring:** on `urlText` change (`.onChange`), if `title` is still empty/untouched, run the pure zero-network parser and populate `title` with the suggestion (host and/or slug). The suggestion is **only auto-applied while the title field has not been manually edited** — once the user types into Title, the URL field no longer overwrites it (track via a `titleWasManuallyEdited` flag, set `true` on any direct edit to the Title field). This mirrors "the user can always overwrite it" (D-02) without fighting the user's keystrokes.
2. **Title** (`TextField("Clip title", text: $title)`) — required, trimmed non-empty guard (matches Rule's title gate). Header: `"Title"`. Below the field, when `trimmedTitle.isEmpty`: hint `"Give this a name to continue."` in `theme.typography.caption` (textSecondary) — verbatim copy reuse from `RuleEditorView`.
3. **Note** (`TextEditor(text: $note)`, `.frame(minHeight: 80)`, optional) — Header: `"Note"`.
4. **Tag** (`TextField("Tag (optional)", text: $tag)`, optional, single free-text field, no autocomplete per D-06) — Header: `"Tag"`.
5. **Domain** (`Picker("Domain", selection: $selectedDomainID)`, `@Query(sort: \Domain.sortIndex)`) — Header: `"Domain"`. Matches `RuleEditorView`'s domain picker exactly (including the `"None"` tag option, though CLIP-03 files strictly by domain — retain the picker's `None` option only if the model allows an unfiled clip; otherwise omit `"None"` and require a domain selection at save. **Executor decides based on the finalized `Clip.domain` optionality** — if `Clip.domain` is non-optional per the planner's schema, drop the `"None"` row and default-select the passed-in domain with no way to clear it.)
6. **Status** (edit mode only, if the planner wants an explicit form control beyond the chip toggle — **optional**, discretion): a segmented `Picker("Status", selection: $status)` with `"Saved"` / `"Acted"` options, OR omit entirely and rely on the chip toggle in `ClipDetailView`/`ClipRow` as the only status-change surface (simpler — **recommended**, avoids duplicate state-change affordances).
7. **Delete** (edit mode only): destructive **"Delete Clip"** row (`theme.colors.danger`), `Button(role:.destructive)`. Confirmation: **at executor/planner discretion per D-11** — given "no stem-style dependents to protect" and a "far less precious" 2-state item (D-05's framing), a lightweight `confirmationDialog` is still recommended for parity with Rules/Collections delete UX (never a silent hard delete), titled `"Delete this clip?"` / message `"This can't be undone."` / confirm button `"Delete Clip"` (danger) / `"Cancel"`.

Field labels `theme.typography.headline` (textPrimary via header styling); values `theme.typography.body` (textPrimary); hints `theme.typography.caption` (textSecondary).

**Save CTA:** `"Add Clip"` (create) / `"Save Changes"` (edit). Toolbar `.confirmationAction` placement, `theme.typography.headline`. Disabled until `trimmedTitle` non-empty AND `urlText` trimmed non-empty (both required per CLIP-02's field list).

**Cancel:** toolbar `.cancellationAction`, `"Cancel"` — dismisses, creates/changes nothing.

### Accessibility (§9.15 — part of "done", applies to all surfaces)

- **Status chip (S1, S2):** `accessibilityLabel` = `"Status: \(statusLabel), \(clip.title)"`. No custom accessibility action needed for a plain two-way toggle (unlike Collections' terminal-clamped chip) — VoiceOver users double-tap to toggle, same as any tappable element; the label updates on next focus to confirm.
- **Tag pill (S1, S2):** `accessibilityLabel` = `"Tag: \(tag)"` when present; the pill is not interactive (no tap target needed beyond the row/detail navigation).
- **Clip row (S1):** `accessibilityLabel`: `"\(clip.title), status: \(statusLabel)\(tagSuffix)"` — e.g. `"Sourdough recipe, status: saved, tag: recipe"`. `.accessibilityElement(children:.ignore)` on the row container with the composed label to avoid VoiceOver duplication from the nested chip.
- **"+" button (S1 header):** `accessibilityLabel`: `"Add clip to \(domain.name)"`.
- **Open Link CTA (S2):** `accessibilityLabel`: `"Open link, \(host)"`.
- **Edit button (S2 toolbar):** default VoiceOver reads "Edit, button" — acceptable (matches `RuleDetailView`, no override needed).
- **Dynamic Type:** no fixed-height text containers anywhere; note content wraps freely; no clipped text except the intentional 2-line title cap on `ClipRow` (matches `RuleRow` precedent).
- **Contrast:** rely on DesignKit token pairings; no accent-on-accent text; tag/caption uses textSecondary on background, which meets AA.

## Open Items for Planner/Executor (Claude's Discretion carried from CONTEXT.md)

These are explicitly left open by 04-CONTEXT.md and are NOT blocking this UI-SPEC — they are scoped decision points the planner/executor resolves within the constraints stated above:
